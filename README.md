# completamente

<!-- Plugin description -->
Code writing helper from local models.
"Completamente" means "completely" in Italian and it's also the union of the words "complete" and "mind".

This plugin provides three features:

- Fill-in-the-middle (FIM) inline completions powered by a local [llama.cpp][3] server.
- Order 89, a code-transformation command that delegates to a user-configured agentic CLI.
- Walkthrough, a step-by-step tour of selected code generated by a user-configured agentic CLI and rendered as inline block-element inlays in the editor.

No cloud APIs, no telemetry from the plugin. Everything you run is under your control.

Inspired by two great projects:

- [ThePrimeagen/99][2] â€” the code transformation workflow behind Order 89

All the good ideas in this code come from these projects, all the bad ideas are my mistakes.

## Installation

The only way to install this plugin is to build it from the command-line.

```
git clone https://github.com/lucatume/completamente.git
cd completamente
./gradlew buildPlugin
```

Then install the plugin in the IDE using `Settings/Preferences` -> `Plugins` -> `Install plugin from disk...`.

## Usage

### Server

The FIM completion side of the plugin needs a local server exposing the llama.cpp `/infill` endpoint. [llama.cpp][3] is the recommended option. For example:

```
llama-server --port 8012 --fim-qwen-30b-default
```

Point the plugin at it by setting the **Server URL** (default: `http://127.0.0.1:8012`) in `Settings/Preferences` -> `Tools` -> `completamente`.

Order 89 and Walkthrough do not use this server. Both shell out to a configurable agentic CLI of your choice â€” see [Order 89](#order-89) and [Walkthrough](#walkthrough) below.

### Configuration

All plugin settings live at `Settings/Preferences` -> `Tools` -> `completamente`. The FIM completion side:

| Setting | Default | Description |
|---------|---------|-------------|
| Server URL | `http://127.0.0.1:8012` | URL of the llama.cpp `/infill` endpoint |
| Context size | `32768` | Total token budget |
| Max predicted tokens | `128` | Tokens per completion |
| Enable auto-suggestions | On | Show FIM suggestions while typing |
| Number of chunks | 16 | Max ring buffer chunks as extra context (0 = disabled) |
| Chunk size (lines) | 64 | Lines per ring buffer chunk |
| Max queued chunks | 16 | Max chunks in the queue |

Order 89 and Walkthrough each have their own CLI command setting â€” documented in their respective sections below.

### Completions

Completions appear as ghost text in the editor. Press **Tab** to accept, or continue typing to dismiss.

The plugin sends a request to the llama.cpp `/infill` endpoint containing:

1. **Current file** -- split at the cursor into prefix and suffix (windowed for large files)
2. **Structure chunks** -- cross-file symbol definitions resolved via PSI references
3. **Ring buffer chunks** -- recent code from files you opened, saved, or copied from

A token budget (`contextSize - nPredict - 20`) ensures the prompt fits. File content gets first priority; remaining budget is filled greedily with structure chunks, then ring buffer chunks.

### Order 89

Inspired by [ThePrimeagen/99][2]. Select text (or place your cursor for insertion), hit `Ctrl+Alt+8` (`Opt+Cmd+8` on macOS), type a prompt, and the selection is replaced with code returned by a configurable agentic CLI.

The plugin builds a prompt containing:

1. **Your instruction** â€” the prompt you type in the dialog
2. **The file under edit** â€” referenced by absolute POSIX path; for files larger than 200k characters only a 50k window each side of the selection is embedded
3. **The selection** â€” its text and its `start_line:start_col`â€“`end_line:end_col` range
4. **Explicit instruction to return the modified selection as a single fenced code block**, never to edit files directly (concurrent invocations on overlapping ranges would otherwise corrupt the buffer through line drift)

The prompt is written to a temp file. The configured CLI is launched with its working directory set to the project root, the temp file path substituted into the command, and its STDOUT post-processed (fenced-block extraction, prose stripping, indentation matching, trailing-newline preservation) before being spliced back into the editor. The temp file is always cleaned up.

ESC during a running invocation destroys the CLI process (force-killed after a 250 ms grace period) and discards the temp file.

| Setting | Default | Description |
|---------|---------|-------------|
| Order 89 CLI command | `pi --tools read,grep,find,ls @"%%prompt_file%%" -p "Execute the instructions in the files"` | Shell-tokenized command. `%%prompt_file%%` is replaced at runtime with the absolute path to the generated prompt file. The process runs with its working directory set to the current project root. |

Any agentic CLI that reads a file path argument and writes the answer to STDOUT will work â€” `pi` is just the default.

The configured command is executed through an interactive shell (`$SHELL -ic`, falling back to `/bin/sh` if `$SHELL` is unset or blank), so PATH and other init from `~/.zshrc`/`~/.bashrc` are picked up just as in a terminal tab. Tools whose PATH setup lives in interactive rc files (`nvm`, `asdf`, `mise`, `pyenv`) work without extra configuration. Login-only files (`~/.zprofile`, `~/.bash_profile`) are *not* sourced â€” if you have PATH exports there, move them to `~/.zshrc`/`~/.bashrc`, or (zsh) into `~/.zshenv` so both interactive and non-interactive shells see them. Standard shell quoting applies â€” wrap paths with spaces in `"..."` and the shell will keep them intact.

### Walkthrough

Select code (or place your cursor on a line), hit `Ctrl+Alt+9` (`Opt+Cmd+9` on macOS), type a question about the selection, and a configured agentic CLI generates a step-by-step tour of the code. Each step is rendered as a block-element inlay above the highlighted range â€” a bordered body with a downward-pointing wedge anchored on the first non-whitespace character of the highlighted line. The wedge X is recomputed on every paint so it tracks scroll, soft-wrap, font-size changes, and edits above the anchor. Chevron buttons inside the inlay (`Â«` first, `â€ą` previous, `â€ş` next, `Â»` last) navigate between steps; a counter shows the current position. ESC closes the active session.

The plugin builds a prompt containing:

1. **Your instruction** â€” the prompt you type in the dialog
2. **The file under edit** â€” referenced by escaped POSIX path; for files larger than 200k characters only a 50k window each side of the selection is embedded
3. **The selection** â€” its text and its `start_line:start_col`â€“`end_line:end_col` range (1-indexed, end column exclusive)
4. **Investigation, focus, depth, narration-style, and rules envelopes** â€” instruct the agent to read related files through its own tools, anchor the first step on the user's selection, fan out to callers/callees/wire-boundary peers, and return its final answer as a single `<Walkthrough>` block of `<Step file="..." range="L:C-L:C">` elements with optional `<Narration>` text

The prompt is written to a temp file. The configured CLI is launched with its working directory set to the project root, the temp file path substituted into the command, and its STDOUT parsed as the `<Walkthrough>` block. Steps that reference files outside the project, missing files, binary files, or out-of-bounds ranges are dropped (with an aggregated notification); a session opens only if at least one step resolves.

Up to 5 walkthroughs are cached per project, persisted across IDE restarts. Cached entries are evicted when any referenced file is modified (mtime mismatch) or removed. Recall a cached walkthrough with `Ctrl+Alt+0` (`Opt+Cmd+0` on macOS); the chooser lists fresh entries newest-first and rehydrates the selected one into a live session without re-invoking the agent.

ESC during a running invocation cancels the in-flight CLI process via the same shell-descendant teardown Order 89 uses (250 ms grace, force-kill, temp file always removed). The temp file is always cleaned up.

| Setting | Default | Description |
|---------|---------|-------------|
| Walkthrough CLI command | `pi --tools read,grep,find,ls @"%%prompt_file%%" -p "Read the instructions in the file and produce the walkthrough"` | Same execution model as Order 89 â€” `$SHELL -ic`, `%%prompt_file%%` substituted at runtime, project root as the working directory. The agent is expected to be read-only; it must not modify any file. |

## Running the FIM server

The FIM completion side of the plugin needs a llama.cpp server reachable on the configured **Server URL**.

```bash
llama-server --fim-qwen-30b-default
```

Order 89 and Walkthrough do not connect to this server, so a single llama.cpp instance with default sizing (the preset's `--fit` auto-sizing) is all you need. If you previously ran a second llama.cpp server for Order 89 you can shut it down.
